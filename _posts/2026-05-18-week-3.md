Three weeks in and the shape of Sunday is settling. Week 1 said the product was the shell, not a window. This week's lesson is the sibling: the *answer* is a layout selected by an engine, not a screen drawn by hand. Most of the visible work landed under that single idea — a typed `LayoutEngine` that takes a content shape (`.image`, `.images`, `.video`, `.text`) and an intent (`.returnToUser`, `.confirm`, `.error`) and picks a layout (Focus, Shelf, Grid, Hero, Card, Feature, Toast, Marquee). The hand-built `LearningView` and `SurfaceView` retired into the engine. So did the second runtime that had been running everything Apple's on-device model couldn't reach.

## packages, finally a contract

The first move of the week was admitting Sunday needed a plugin shape. Until Monday, every external image source — local SDXL, ImagePlayground — lived inside `CanvasState` directly. Adding Unsplash by writing more `CanvasState` was the obvious version and the wrong version. The right shape is `BackgroundSource`, a four-method protocol in a tiny `SundayPackageCore` Swift package:

```swift
public protocol BackgroundSource: Sendable {
    static var identifier: String { get }
    static var displayName: String { get }
    static var supportedStyles: Set<BackgroundStyle> { get }
    func image(for query: String, style: BackgroundStyle) async throws -> BackgroundResult
}
```

One call, one result. No batching, no streaming, no twelve-options grid — the canvas only shows one background at a time. Errors are typed so the core wraps them into plain language at the boundary (*"the photo service isn't configured"*, not error code 401). Attribution is a first-class field on `BackgroundResult`, because Unsplash's license requires it and any future provider with the same constraint shouldn't have to invent its own.

Unsplash landed as the first real implementor. Pexels followed in the same shape. Then YouTube — but YouTube is video, and the result shape is genuinely different (provider + video ID, not a local file URL), so a second protocol `VideoBackgroundSource` joined the package core. Two protocols rather than one polymorphic blob: each pays for its own existence with a concrete shape, and most photo packages will never want a video method.

The packages are Swift Packages on disk, in a `Packages/` directory, depended on by the main app. They build and test in isolation. They know nothing about Sunday's internals. The first-party API-key prompts use the same nudge ("set my YouTube API key to …"), the same Keychain-backed storage, the same plain-language wrap. WeatherPackage landed as the proof that this shape works for an Apple framework, too — WeatherKit + CoreLocation + MapKit, wrapping the former `WeatherFetcher` actor, with Apple's required "Apple Weather" attribution riding every reply through `DataAttribution`.

The architecture decision the protocol made explicit: **packages are passive data fulfillers; matching, dispatch, and final phrasing stay in Sunday core.** A package never says "I think this query is for me" — `CapabilityRouter` does. A package never decides what surface to mount — `LayoutEngine` does. The package is the source of the data only.

## the layout engine retires the hand-built surfaces

The second move of the week was the one that's most visible to the user, and the one that took the most retiring. Sunday had two parallel rendering paths: `RenderSpec` → `SurfaceView` for text replies, and `LearningSurface` → `LearningView` for the explain/show flows. Both had grown sophisticated over five weeks of iteration. Both were going to need to grow further. Maintaining them in parallel with the new `LayoutEngine` was the wrong floor.

The engine is one decision: given a `CapabilityResult` (image, images, video, text) and a `LayoutContext` (intent, optional caption), return a `LayoutSelection { layoutId, content }`. The catalog of layouts now covers the common shapes:

- **Toast** — short confirmations. *"Saved to Photos."*
- **Card** — title + body + attribution. *"Tell me about photosynthesis."*
- **Focus** — single image + caption. *"Show me a photo of mount fuji."*
- **Feature** — text + image editorial pair. *"How does rain work?"*
- **Shelf / Grid** — collections of images, picked by count (≤4 / 5+). *"Show me photos of forests."*
- **Hero** — single embedded video + caption. *"Show me a video of california."*
- **Marquee** — display moment. End-of-onboarding *"Hello, [name]."*

Migration happened in slices. Focus landed first — *show me a photo of X* split off `handleShowLearning` into `handleShowPhoto` and routed through the engine. Shelf and Grid landed bundled because they share a matcher and only differ by the count threshold inside `LayoutEngine.select`. Hero landed for *show me a video of X*; *play me a video of X* stays on the existing fullscreen overlay because its audio handoff and multi-candidate retry are delicate enough that migrating them is meaningful work for no behavior change. "Play" and "show" are now genuinely different verbs with genuinely different surfaces.

`LearningView` retired Thursday. The explain-with-illustration path moved to Feature; the explain-without-illustration path moved to Card; the show-mode-illustration path moved to Focus, matching the photo show-mode path that had already migrated. The grounding attribution (Wikipedia source URL + article title) flowed through a new `textAttribution: DataAttribution?` field on `FeatureContent` so the editorial pair could credit Wikipedia for the body while crediting Sunday for the sketch. The reading is *"by Photosynthesis on Wikipedia"* — slightly off grammatically because the existing `AttributionLine` builds from `authorName + sourceName`, but grammatical enough until a second variant earns its place.

`SurfaceView` retired Friday. Sixty-eight call sites stayed working by turning `surface(_ spec: RenderSpec)` into a thin adapter: if the spec has attribution or a detail body, build a Card; otherwise build a Toast; either way set the mode to `.layout(selection)` and honor the spec's existing `SurfaceRecede` timing. The adapter is the one piece worth keeping because the existing producers have meaningful recede choices (info 8s, complex 12s, reference 20s) that the layout's static dismiss timing couldn't yet express. `RenderSpec` and `GroundingAttribution` survive as input DTOs; a renaming pass to a proper `surfaceToast(_:)` / `surfaceCard(_:)` API is a follow-up.

What this consolidation buys is mundane and load-bearing: future polish (a new transition, a new attribution variant, a new sizing rule) lives in one place. Hand-built parallel surfaces were a tax we were paying every time we touched a flow.

## the brain that retired

The third move of the week wasn't planned to go this far. Sunday started the week running two brains: Apple's Foundation Models framework (≈3B on-device, sub-second, used for classification and structured output through `@Generable`) and Ollama serving `qwen2.5:7b` over `localhost:11434` (≈3-5s, used for tool-calling Q&A). The split was real and load-bearing — small model for the fast path, larger model for reasoning. That's the architecture the spec defended in week 1.

Then we read WWDC 2025 properly. MLX-Swift can do everything Ollama was doing for Sunday: Hugging Face model load, in-process inference, KV cache for multi-turn, quantization, even LoRA adapter fine-tuning. A native Swift API, no separate user install, no `localhost:11434`. The week was supposed to be a *side-by-side comparison* through Phase 4 of the audit plan: ship MLX behind a `UserDefaults` flag (`sunday.deepBrain.engine`), run both routers through a real eval corpus, decide on data.

The eval data decided harder than expected. MLX with `qwen2.5:7b-instruct-4bit` matched Ollama on tool-call success, beat it on grounding accuracy for the RAG corpus, and ran in roughly the same wall-clock time on the same hardware. Removing Ollama wasn't just a flag flip — it was a subtraction. So Phase 4 went further than the plan envisioned: Ollama is gone. The router, the engine adapter, the install card check, the legacy flag, the `OllamaRouter.swift` file, the `ImagePromptEnhancer` `localhost:11434/api/generate` call — all out. Sunday is now all-in on Apple's ML stack: Foundation Models for the sub-second classification path, MLX-Swift for the tool-calling reasoning path, both running in-process, both pulling from the same `ConversationStore` and the same `RecallService` over `NLContextualEmbedding` vectors. The `Tools/` directory's actual implementations didn't change — only the *registration* (FM's `@Generable Arguments` for one router, MLX's prompting conventions for the other).

The unblocked Phase 3 work — Foundation Models features Sunday wasn't using — followed in the same week. `PartiallyGenerated` streams the `LearningExplainer` body sentence by sentence with the existing fade-in motion. `IncludeSchemaInPrompt: false` was already set on the heaviest `Generable` site; prewarming joined it at the call sites where an intent looked deep-brain-shaped. The Foundation Models Instrument trace now shows prewarm hits before submit. Same model, less waiting.

## the marquee moment

The smallest user-visible change was the most felt. End-of-onboarding had been a 0.6-second black flash between the last wake-word enrollment recording and the ambient canvas. The new shape is a Marquee that holds for ~1.8 seconds: the user's entered name in `.sundayDisplay`, no subheadline, then a fade to ambient. *"Hello, Joe."* The name is the one thing Sunday now knows that the OS couldn't give it, and it's the first thing the system says when it finishes setting itself up.

`MarqueeLayout` got one calibration in the slice — the subheadline rose from 20pt to 32pt Light so a 72pt headline with a 20pt subhead stopped feeling lopsided. `WelcomeView` migrated onto MarqueeLayout the same morning. Same look, one fewer hand-rolled moment. A `surfaceMarquee(headline:subheadline:)` producer landed on `CanvasState` so future "welcome back" and "all set" beats become a one-liner whenever they earn their place.

## next

The SundaySessionKit spec landed mid-week — a protocol extension for packages that need multi-turn state (a trivia quiz, a recipe walked step by step, a workout interval timer, a morning briefing). Two interaction modes: *prompted* (Sunday waits for a user response and the package evaluates it) and *narrated* (Sunday waits for a navigation command and the package advances). The protocol lives in a new `SundaySessionKit` Swift package depending on `SundayPackageKit`. The reference packages — `TriviaPackage` (prompted) and `RecipePackage` (narrated) — are the first build targets, alongside the six new session layouts (Quiz, Flashcard, WordGame, Scoreboard, Step, Briefing) that join the catalog. None of this is in main yet. The spec is the load-bearing first move.

After session-kit, the package backlog is concrete: Maps (MapKit + CoreLocation for *directions to* and *how long to drive to*), Messages and Mail (Apple frameworks via AppleScript bridge for send; read deferred for TCC reasons), Apple Music and Spotify competing on user preference for `audioPlayback`, Giphy for animated `inlineImage`, and a read-only Apple Photos package that lets Sunday query the user's own library. Each one follows the same hub-and-spoke pattern Weather and Unsplash already proved: a `Packages/<Name>/` Swift Package, a `SundayPackage` conformance, registration with `CapabilityRouter.bootstrap`, matching logic staying in Sunday core.

The wake-word per-user fine-tune through `MLUpdateTask` is still the slice that unblocks the always-on listener. It didn't ship this week. It is, again, the next slice.
