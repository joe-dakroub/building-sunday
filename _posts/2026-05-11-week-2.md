The work this week kept arriving at the same shape: build the obvious version of a feature, watch the eval data or the feel of it show the obvious version was wrong, find the right version was a primitive Apple had already shipped. Three big examples, a few small ones. The clearest sign you've understood a platform is how much of the work it does for you when you let it.

The Card concept retired on Tuesday. Cards came from the Tauri prototype that had no system access — they were Sunday's own little database of *everything you'd asked it to remember*. On macOS, EventKit, Contacts, PhotoKit, FileManager + Spotlight, HealthKit, and MusicKit are already authoritative. Replicating them as Cards is a sync problem with no information gain. What still needs to be Sunday's, because Sunday creates it, is the conversation transcript. So Cards came out of the schema and `Conversation` + `MessageRecord` went in — proper SwiftData models, queryable, observable. A one-time migration from the old JSON blob runs at launch and renames the file `.migrated` rather than deleting it. Persistence is still a Sunday principle. The data layer is now honest about what is mine and what is macOS's.

## the calendar and reminders loop

The biggest concrete output of the week was closing the read/write loop on both productivity surfaces. Reminders read landed Tuesday morning — *"what do I need to do today"*, *"anything urgent"*, *"what's flagged"* — aggregating across lists and surfacing the list name as a tag suffix per item. Reminders write followed in the afternoon: add and complete. *"Remind me to buy milk tomorrow"* uses `NSDataDetector` to find the date in the title, strips that span, capitalizes the rest, and saves to the user's default list. Each confirmation echoes the parsed result back so voice mistranscriptions are visible. Then snooze, reschedule, calendar add, and delete in single slices each — by Tuesday evening, both Reminders and Calendar speak read, add, complete, snooze, reschedule, and delete, with a shared dispatcher that tries reminders first and falls through to calendar when delete or reschedule misses.

The natural-result polish was the day's quietest fix and the one that made it sound human. *"How many days until July 4, 2026"* became *"How many days until July 4"* — a one-line addition to the system prompt telling the model to omit the year when it matches the current one. And `CalendarCapability` learned to count and group: *"How many standups this week"* now returns *"You have 4 standup meetings this week: Monday–Thursday."* Consecutive days collapse into a range. The user reads the answer instead of counting items in a list.

There is also an Identity capability now. *"Who am I?"* / *"What's my number?"* / *"Where do I live?"* read the macOS user's Me card via the Contacts framework, with `NSFullUserName()` as the name-only fallback. Empty states never name the framework — they point at the exact place to fix the gap: *"I don't see your contact card. Open Contacts and set yourself as the Me card."* Every preference is a sentence; every gap is, too.

## the deleted slice

Wednesday's larger story is what got thrown out. The plan was to replace the English regex matchers with an Apple Foundation Models intent extractor — `@Generable` structs, the System Brain reading a question and emitting a typed `RemindersIntent` or `CalendarIntent`. The point was multilingual: regex is English-shaped; model intent extraction speaks every language Apple's model speaks. We built it. `IntentMatcher`, four `Model*Matcher` files, seventy-six unit tests on the mappers, an eval harness.

The eval data killed it. On rich schemas — a 13-case enum with associated values, or a flat 11-field struct — Apple's on-device 3B model takes sixty seconds per call and misclassifies ambiguous queries as null. A 7B Ollama model would help accuracy but stays at three to five seconds per call. Either way, model intent extraction puts a multi-second floor under the common path. That is the wrong floor.

The right question came mid-week: if regex is so fast, why not use it for all languages? Apple's `Translation` framework, which we hadn't considered, turned out to be the answer. We probed it. Forty-one milliseconds on first call, sixteen to eighteen milliseconds cached, exact-match translation of imperative French into the English the regex already catches. Sub-millisecond language detection through `NLLanguageRecognizer`. On-demand pack install, supported-language list grows with macOS itself. The architecture is now three layers, each on its strength: regex in microseconds for English imperatives, Translation in tens of milliseconds for non-English imperatives, Apple FM in seconds for free-text Q&A in any language Apple's model speaks. Zero seconds of model-intent tax on the common path. The model-matcher work came out of the repo on the same commit that this plan went in.

The other discovered primitive of the week was `NLContextualEmbedding` — 512-dimensional transformer embeddings on-device, multilingual by default, in the OS since macOS 14. Vector recall over conversation messages, at Sunday's actual scale of thousands of vectors and not millions, doesn't need `sqlite-vec` or a separate database. The embedding bytes go on the `MessageRecord` itself as `Data`; cosine similarity in Swift runs over the result of a SwiftData fetch. A few milliseconds across ten thousand messages on Apple Silicon. The original tracker memory had `sqlite-vec` plus `nomic-embed-text` as the candidate stack; the platform's own embedding model and SwiftData's own storage did the job with no new dependency.

## first launch

Onboarding is what a fresh install of Sunday now sees: a quiet *Hello, I'm Sunday.*, a single name field, then three recordings of the wake word — *Sunday* spoken three times, 1.5-second windows, captured as 16 kHz mono Float32 WAV files under the app's container. The recordings feed the per-user fine-tune that will activate the always-on wake-word listener; until that fine-tune lands the listener stays dormant. The data model now distinguishes Admin (the person who set Sunday up) from Guests (others the Admin introduces) — only the Admin path enrolls in v1. Sunday recognizes its existing Admin on launch and skips the flow entirely. The design language is the same true black, white SF Pro, fade-not-slide as the rest of the shell. There is no separate onboarding aesthetic, because there is no separate onboarding product.

There is also a test bundle now — Swift Testing, async-friendly, parallel by default — covering the durable helpers: token matching, day-run grouping, duration parsing, identity date math, Oxford-comma joining. About forty tests, all deterministic, no EventKit, no network. The parser layer is intentionally untested because it was about to migrate; that migration didn't go the way the plan expected, but the tested helpers are the ones we kept.

## the shadow exception

The last subtraction of the week was the canvas scrim. Sunday paints generated backgrounds behind the surface in some moods, and a 45% black overlay sat on top of the image to guarantee white type readability. It worked, and it muted every image into the same drab field. The image is the user's chosen environment, and muting it costs the feeling that drew them to it.

The fix amended the spec. Week 1's design rules said *shadows → never*. The new rule is *shadows → legibility only, over image and video backgrounds, never on solid color, never as decoration*. A single design primitive, `LegibilityShadow`, compounds a tight crisp drop (radius 4, opacity 0.55) with a wider soft lift (radius 12, opacity 0.30). Applied per-surface, conditional on whether the environment is an image. The backdrop stays vibrant. The type still lifts. One rule changes; the principle behind it — *every visual decision earns its place* — does not.

## next

The wake-word reactivation is now unblocked on the data side — three enrollments per Admin land on disk during onboarding — but blocked on the per-user fine-tune via `MLUpdateTask`, which is the next slice. Speaker fingerprinting (MFCC plus cosine, for the Admin/Guest distinction) follows that, then Guest enrollment, then the proactive insight surface that's still on the original spec but still hasn't been built. RAG over conversation history through the new vector store, wired into prompt assembly, is the small follow-on once the user has a reason to invoke it.
